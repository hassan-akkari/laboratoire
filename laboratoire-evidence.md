# laboratoire — CV Evidence (mined & adversarially verified)

> **What this is.** An evidence-grounded inventory of `laboratoire` (solo pnpm + Turbo monorepo) for CV use. Every claim below was read out of the actual source, then a second pass of skeptical agents re-opened the cited files to confirm or refute. Complexity ratings are **post-verification** (1 = trivial/styling, 5 = genuinely hard). Where the second pass corrected the first, both numbers are shown (e.g. `3 → 2`).
>
> **Method:** mapped the repo (154 commits, ~15.8k LOC TS/TSX, 4 workspaces), read the non-trivial source per app/package, used git to separate hand-written from generated/library code, then ran an adversarial verification pass over the 29 strongest/most-inflation-prone claims. Generated `2026-06-16`.
>
> **Three honest headlines, read these first:**
> 1. **The "same booking domain across two paradigms" angle does not hold.** `apps/web-next` has the full booking/quote/checkout/order/admin domain. `apps/web-react` has **one** endpoint — `GET /api/ping → {ok:true}` — and zero quote/cart/checkout/order/pricing/guest code. Do **not** claim a parallel domain implementation; web-react is a stack-wiring scaffold (see §B).
> 2. **The real engineering is concentrated in `apps/web-next` (backend/server-side).** Portfolio (`docs`) is mostly polished static content; `packages/ui` is the weakest surface (wrappers + a copy-pasted commercial kit).
> 3. **`CLAUDE.md` is partly stale.** It describes web-next as in-memory-only with a `middleware.ts` static-cookie gate. Reality on this branch: a real **Neon/Drizzle Postgres** layer exists for leads/users/site_config, the middleware is now `proxy.ts` (Next 16 rename), and the "dead web-react alias" gotcha was already fixed. Verify doc claims against code before citing.

## A. The 3–4 strongest things (worth CV space)

**1. Cal.com webhook HMAC verification — and the bare-hex interop fix.** *(Complexity 3, low inflation, the clearest single piece of hand-engineering in the repo.)*
A timing-safe HMAC-SHA256 webhook verifier where the real story is a prod-breaking bug correctly root-caused: bare-hex digest vs the common `sha256=`-prefixed convention, compounded by `timingSafeEqual` throwing on unequal buffer lengths and silently failing closed — so every live webhook 401'd. Diagnosed against upstream source, fixed in impl + test, with raw-body-before-parse ordering preserved.
- *Rejected alternative:* blindly following the GitHub/Stripe `sha256=`-prefix convention (exactly what shipped first and broke it) — the lesson is "verify the digest format against the sender, don't assume the ecosystem default".
- *What I'd do next:* add a timestamp/replay window and signing-secret rotation; the current verifier authenticates the body but not freshness.

**2. Server-side correctness of the booking backend: idempotency + zero-client-trust + money-math ordering.** *(Complexity 3, med-but-honest inflation.)*
Three things that together read as senior judgement: an idempotent `processCheckout` with TTL + cap eviction + orphan-key GC; checkout that **re-runs the quote server-side and never trusts the client total**; and deliberate fee/tax/discount sequencing with epsilon rounding.
- *Rejected alternative:* trusting the client-submitted price/quote (faster, common in demos) — rejected in favour of server recompute.
- *What I'd do next:* the Neon/Drizzle DB already exists for leads — **persist orders to it** and back idempotency with a DB unique constraint, closing the in-memory single-instance gap (currently `MVP-ONLY`).

**3. A coherent, test-covered security slice: iron-session admin auth + origin/CORS allowlist + Neon/Drizzle schema.** *(Complexity 3, low inflation.)*
Sealed-cookie admin sessions (secret-length guard, secure flags, getter-vs-guard split, defense-in-depth presence/seal split), env-driven public/admin origin isolation with production-safe Vercel-preview reconstruction, and a Postgres schema with a partial unique index (webhook idempotency) + singleton CHECK + a typed-null `DEV_NO_DB` escape hatch.
- *Rejected alternative:* hand-rolling session crypto / a JWT scheme — chose `iron-session` and wired it correctly instead of inventing crypto.
- *What I'd do next:* multi-user/RBAC + password reset; close the unknown-user-vs-wrong-password timing seam the verifier flagged.

**4. Monorepo build/CI/tooling — including the AI-augmented dev workflow.** *(Complexity 2–3, low-med inflation.)*
A secretless, cache-aware CI gate that reasons about mocked DB + source-alias resolution (and replaced a broken Vercel `tsc` check), pnpm/Turbo install hardening, and a bespoke Vite source/dist UI resolution gate. Distinctively, the repo is operated through a hand-curated AI-governance layer (`CLAUDE.md`/`AGENTS.md`/bootstrap/debt ledger) with a "forbidden-first" agent design and lockfile-exact version discipline.
- *Rejected alternative:* relying on Vercel's default per-app checks (the bare-`tsc` Typecheck that was already broken with no root tsconfig).
- *What I'd do next:* pre-commit hooks (no local gate today) + remote Turbo cache; reconcile the non-uniform TS strict flags (tracked, still open).

## B. Do NOT claim / thin spots

- **"Same booking/checkout domain across two paradigms (Next.js vs Redux/Vite)."** ❌ **False.** `web-react` has no booking domain — one `GET /api/ping`. It is a stack-wiring scaffold (`BrowserRouter` with zero routes, RTKQ with one boolean query, `NextSteps` is a literal TODO list). Frame web-react honestly as *"a minimal Redux Toolkit + RTK Query + MSW reference scaffold,"* not a parallel implementation.
- **`packages/ui` as a "design system / component library."** ❌ 3 one-line HeroUI wrappers + a theme toggle + a **copy-pasted commercial Catalyst kit** (25 of 27 files dead, the other 2 unused) + default Storybook. The Catalyst components are **not authored work**.
- **The docs Redux/RTK Query layer as the app's live data backbone.** ❌ The home page renders from static `data/*.ts`; one endpoint is dead code; the `/cv` fetch sits behind a hardcoded fallback. It's a *showcase* of the pattern, not the data path.
- **Persistent / distributed order storage.** ❌ Orders are in-memory on `globalThis`, lost on restart, single-instance, `MVP-ONLY`. (The Neon DB persists **leads/users/site_config only**.)
- **Booking-side authentication.** ❌ The `/checkout` cookie is a plaintext sentinel (`"active"`), no crypto/CSRF, fake by design. Only the **admin** side is real auth.
- **A "pricing engine."** ❌ Fixed 2-promo table, 2 models, single EUR, fixed 8%/12% constants.
- **An "automated i18n pipeline."** ❌ Three hand-maintained JSON files; no codegen, no parity test.
- **"Enterprise CI/CD" / "distributed caching."** ❌ One ~76-line workflow on a feature branch; local/CI filesystem cache only; no deploy job (Vercel's git integration handles deploy).
- **Uniform TS strictness "everywhere."** ❌ web-next/ui carry only `strict:true` and miss the extra flags docs/web-react enforce; `verbatimModuleSyntax` differs. (Tracked as A12.)
- **AI-tooling *document volume* as engineering.** ❌ Scaffolding + prose are model-generated; the genuine contribution is curation, invariants, and accuracy.
- **`orchestrator` as a "multi-agent runtime."** ❌ It's Markdown/JSON config orchestrating third-party skills; never verifiably executed.
- **Dormant code:** `ContactForm`/`AboutSection`/`HighlightsSection`/`PortfolioSection`/`RoadmapSection`/`ContactSection` in docs are defined but unrendered; `getGithubProfile` unused. Don't present as shipped.
